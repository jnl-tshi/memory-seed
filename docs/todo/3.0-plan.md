# Memory Seed 3.0 - Plan

> **Status: REVIEWED + DECIDED (2026-06-15).** This plan folds in the review of the
> existing code, the multi-user proposal, the Memory Explorer research report, and the user's latest
> decisions: Pillar B will be a separate install from the core `memory-seed` package; new entry IDs
> will widen, but to an 80-bit v2 format rather than the research draft's full 128-bit format; and
> related-entry linking becomes a first-class part of the graph model.

## Context

3.0 is the line where Memory Seed stops being purely solo-and-agent-facing. Two researched themes
define it, and they share one technical spine:

1. **Finish multi-user session memory.** Phase 1 (read-only dual discovery, shipped 2.9.0) and
   Phase 2 (opt-in user-aware write targets + hooks, shipped 2.10.0) are done. A-P3 integrity
   validation, A-ID entry-ID widening, A-P4 MCP metadata/filters, S2 participant registry reading,
   and A-P5 migration are implemented in the current unreleased worktree. What remains is the
   Explorer companion package.
2. **Create a human-facing Memory Explorer.** A local, read-only UI over the same retrieval the MCP
   server uses: same answers as MCP, richer navigation for humans.

Non-negotiable constraints:

- Markdown remains the source of truth.
- Generated indexes are gitignored and rebuildable.
- No SQLite database is committed as source of truth.
- Because this repo lives under OneDrive, any optional cache should prefer per-file, content/mtime
  keyed artifacts over a monolithic SQLite database.
- Existing `entry_id` and `chunk_id` retrieval contracts keep working.
- Per-user filenames are attribution, not privacy or access control.

## Review Correction: Retrieval Is Already Mostly Shared

The original draft made "S1 - extract a shared retrieval module" the first increment. Code review
showed that the genuinely shared part already exists: `semantic_cache.py` holds the parser
(`extract_memory_chunks`, already dual-reading both layouts through `iter_session_documents`) and the
ranker (`rank_memory_chunks` / `rank_session_memory`). `mcp_server.py` is a thin wrapper over that
logic.

The only still-MCP-coupled piece is the search orchestration and result-dict contract:
`call_tool`, `format_search_results`, `_ranked_to_dict`, `_chunk_to_dict`, and `_semantic_provider`.
That extraction has one real second consumer: the Explorer. Therefore it travels with Pillar B, where
the companion package validates the boundary. A-P4 can edit the MCP result contract in place.

## Pillar A - Complete Multi-User Session Memory

### A-P3 - Integrity Validation

Implemented as 2.12.0, but not released at this plan update.

`memory-seed links check` validates both layouts:

- duplicate `entry_id`
- duplicate per-user file `hash_id`
- dangling `related_entries` / `related_memories`
- malformed per-user frontmatter
- invalid user slugs
- filename-to-frontmatter user/date mismatch
- unsupported `schema_version`

The command exits non-zero for CI. `doctor` only surfaces a non-fatal one-line summary that points to
the dedicated command.

### A-P4 - MCP Metadata + Filters

Implemented in the current unreleased worktree.

Add the missing multi-user metadata to `memory_search` / `memory_get_chunk` without changing existing
chunk IDs:

- `file_hash_id`
- `path` / `source`
- `user`
- `session_date`
- `related_entries`

Add filters:

- `user`
- `date_from`
- `date_to`

Compatibility rules:

- Existing `ms-xxxxxxxx` entry IDs remain valid chunk IDs.
- Section chunk IDs keep their existing `<entry_id>#<section-slug>` shape.
- Legacy flat files have `user = null` until S2 maps `user_initials` to participant slugs.
- Per-user files derive `user` from the date-directory filename stem, with frontmatter used for
  validation and display.

Implementation:

- Extend `MemoryChunk` in `memory_seed/semantic_cache.py` with file-level metadata.
- Pass `SessionDocument` metadata from `iter_session_documents()` into extraction.
- Filter before ranking when filters are supplied, so `top_k` applies to the filtered corpus.
- Update `memory_seed/mcp_server.py` schema and result dictionaries.
- Add tests in `tests/test_semantic_cache.py` and `tests/test_mcp_server.py`.

### S2 - Participant Registry in Project YAML

Implemented in the current unreleased worktree.

`.memory-seed/project.yaml` already stores agent selection. Add participant registry keys alongside
the existing agent-selection keys without disturbing the existing parser:

```yaml
agents:
  - codex
  - claude

project_id: memory-seed
participants:
  - slug: jean
    initials: JN
    display_name: Jean
```

This registry becomes the source of truth for:

- migration from legacy flat logs to per-user files
- legacy `user_initials -> slug` mapping
- Explorer contributor pages
- future identity display

It is not an auth system.

Current implementation provides a fail-open `read_project_participants()` helper that parses valid
participant entries while preserving existing `agents:` behavior. A-P5 uses these entries for the
legacy `user_initials -> slug` migration map.

### A-P5 - Migration Command

Implemented in the current unreleased worktree.

`memory-seed migrate sessions-layout` includes:

- `--dry-run`
- backups
- idempotence
- mixed-layout safety
- legacy flat-file parsing entry by entry
- `user_initials -> slug` mapping via `.memory-seed/project.yaml`
- preserved existing `entry_id`
- one file-level `hash_id` per output per-user file
- no overwrite of existing per-user files unless the merge is provably safe
- flat sources removed after a successful backup+split so dual-read does not create duplicate IDs

Legacy dual-read stays permanently supported after migration.

## Entry IDs and Related Links

### Decision

Apply entry-ID widening during 3.0, but do not use full 128-bit visible IDs.

Current IDs use `ms-` + 8 hex characters, about 32 bits. Birthday-collision math makes that too small
for team-scale history:

| Bits | Visible form | Collision probability at 10k | 100k | 1M | 10M |
|---:|---|---:|---:|---:|---:|
| 32 | `ms-` + 8 hex | 1.16e-2 | 6.88e-1 | ~1 | ~1 |
| 64 | 13 Base32 chars | 2.71e-12 | 2.71e-10 | 2.71e-8 | 2.71e-6 |
| 80 | 16 Base32 chars | ~0 | 4.11e-15 | 4.14e-13 | 4.14e-11 |
| 128 | 26 Base32 chars | ~0 | ~0 | ~0 | ~0 |

The right tradeoff for this application is **80 bits**: short enough to be readable in logs and links,
but with enough headroom that even 10M generated entries remain negligible-risk. Full 128-bit IDs are
excellent for opaque machine IDs, but overlong for a human-facing session-memory identifier.

### New ID Shape

```text
mse_<16 lower-case base32 chars>
```

Generation:

1. Canonicalize the same metadata currently used by `generate_session_entry_id()`.
2. Hash with SHA-256.
3. Truncate to 80 bits.
4. Encode as lower-case Base32/Crockford-style text.
5. Prefix with `mse_`.

Rules:

- `ms-[0-9a-f]{8}` remains valid forever.
- New generated IDs use `mse_`.
- Migration never rewrites existing `ms-` IDs.
- `memory_get_chunk` resolves both old and new entry IDs.
- `links check` accepts both namespaces and detects duplicates across the whole corpus.
- A write path that can see existing entries should fail loudly on collision instead of silently
  generating a surprise alias.

### Related Entry Linking

Use two layers:

1. **Entry-level links are canonical graph edges.** Add `related_entries` to each entry YAML block:

   ```yaml
   entry_id: mse_0123456789abcdef
   related_entries:
     - ms-8e44a1c7
     - mse_abcdef0123456789
   ```

2. **File-frontmatter links remain coarse session/file relationships.** The existing A-P3
   `related_entries` / `related_memories` frontmatter support stays valid, but it is not the primary
   Explorer graph edge.

A-P4 exposes entry-level `related_entries` in `memory_search` results. `links check` validates both
frontmatter-level and entry-YAML related refs. The Explorer graph should rank explicit entry links
above derived relationships such as same day, same tag, same project, same contributor, or same file.

## Pillar B - Memory Explorer as a Separate Install

### Decision

Create the Explorer as a companion distribution, not as a required part of the core package.

Recommended package split:

- Core package: `memory-seed`
- Explorer package: `memory-seed-explorer`
- Explorer command: `memory-seed-explorer`

The Explorer package depends on `memory-seed`, but the core package must not import Explorer code or
depend on Explorer-only dependencies. A future `memory-seed[explorer]` convenience extra may point to
the companion package, but the primary release unit should remain separate so the core stays small,
local-first, and easy to install for agents and CI.

This matches Python packaging's optional dependency model while avoiding the hidden cost of making a
web stack part of the default install. See:

- Python packaging optional dependencies:
  <https://packaging.python.org/en/latest/specifications/pyproject-toml/#dependencies-optional-dependencies>
- VS Code webviews, which are powerful but resource-heavy and best used deliberately:
  <https://code.visualstudio.com/api/extension-guides/webview>
- Obsidian community plugin security posture:
  <https://help.obsidian.md/community-plugins>
- Tauri architecture for a later lightweight desktop shell:
  <https://v2.tauri.app/concept/architecture/>

### Product Surface

Build the local web app first:

- Python backend exposes a localhost read-only API.
- Static React/TypeScript UI is bundled with the Explorer package.
- The API calls the same parser/ranker/result service used by MCP.
- The initial product opens in the user's browser.

Desktop packaging is a later shell around the same app, not a separate architecture. If a desktop app
is needed, prefer Tauri first because it can wrap a web UI with a lighter native-webview model than a
bundled-Chromium Electron app. Electron remains an option if packaging speed and ecosystem maturity
matter more than footprint.

Obsidian should be inspiration and a later integration, not the first delivery vehicle. The reasons
from the research report still hold: an Obsidian plugin would either duplicate parsing/ranking in
TypeScript or bridge awkwardly into Python, and it inherits a broad plugin trust model. VS Code is a
better second shell for developers because it can reuse the same web bundle/API.

### B1 - Localhost Read-Only API

Package: `memory-seed-explorer`.

Command:

```bash
memory-seed-explorer --cwd . --host 127.0.0.1 --port 0
```

Endpoints:

- `GET /api/search`
- `GET /api/chunks/{chunk_id}`
- `GET /api/timeline`
- `GET /api/graph`
- `GET /api/contributors`
- `GET /api/stats`

Hard rules:

- Bind to localhost by default.
- Read-only in 3.0.
- No writes to session files.
- Retrieval parity with MCP fixtures.
- No persistent index in the first API increment.

### B2 - Read-Only Web UI

Views:

- Search
- Reader
- Timeline
- Graph
- Contributors
- Stats

The primary object is the chunk. The canonical deep link is `chunk_id`.

### B3 - Explainability

Expose:

- matched terms
- matched fields
- lexical score
- semantic score if available
- recency multiplier
- related-entry links
- derived graph edges

### B4 - Optional Cache

Add only after real usage demonstrates need. If added, the cache is:

- gitignored
- rebuildable
- keyed by file mtime/content hash
- never authoritative
- safe for cloud-synced workspaces

### B5 - Curation and Collaboration

Post-3.0. Keep editing and collaboration out of the read-only Explorer MVP. Later curation should use
separate annotation/patch records rather than silently rewriting session history.

## Recommended Sequencing

Do not build a mega-branch. Ship one reviewable increment at a time:

1. **Stabilize the current worktree**: commit and release the already-implemented A-P3, A-ID, A-P4,
   S2, and A-P5 work, or intentionally keep it unreleased before stacking more code.
2. **B0**: scaffold `memory-seed-explorer` as a separate package with dependency isolation tests.
3. **B1**: implement localhost read-only API with MCP parity tests.
4. **B2/B3**: add the human UI and explainability.

```mermaid
flowchart LR
  P3["A-P3 integrity validation (2.12.0 unreleased)"] --> ID["A-ID 80-bit mse_ IDs (current worktree)"]
  ID --> P4["A-P4 MCP metadata + filters + related refs (current worktree)"]
  P4 --> S2["S2 project.yaml participants (current worktree)"]
  S2 --> P5["A-P5 migrate sessions-layout (current worktree)"]
  P5 --> B0["B0 memory-seed-explorer companion package (next)"]
  B0 --> B1["B1 localhost read-only API"]
  B1 --> B2["B2 read-only web UI"]
  B2 --> B3["B3 explainability"]
```

## Risks

- **Release stack risk:** 2.12 is implemented but unreleased; new 3.0 work should not obscure that
  boundary.
- **Identifier contract risk:** widening IDs touches logs, validation, MCP chunk IDs, and graph links.
  Mitigate by accepting both ID namespaces permanently and never rewriting legacy IDs automatically.
- **Retrieval drift:** Explorer must not fork parsing/ranking from MCP. Mitigate with parity tests.
- **Dependency creep:** Explorer dependencies stay in `memory-seed-explorer`, not core `memory-seed`.
- **Cloud-sync cache risk:** avoid authoritative SQLite or monolithic cache files in OneDrive paths.
- **Governance creep:** editing/collaboration stays post-MVP.

## Definition of Done

Each increment ships with green tests, `doctor` healthy, docs updated, and a concise session log entry.

Specific gates:

- A-ID: old `ms-` IDs and new `mse_` IDs both validate, resolve, and remain fetchable.
- A-P4: metadata appears in MCP results; filters run before ranking; existing chunk IDs still fetch.
- S2: existing agent-selection `project.yaml` behavior is unchanged when participant keys are present.
- A-P5: dry-run, backup, idempotence, blocked ambiguous mappings, and mixed-layout tests pass.
- B0/B1: installing `memory-seed` alone does not install/import Explorer dependencies; installing
  `memory-seed-explorer` exposes the read-only command; API search/fetch matches MCP fixtures.
- B2/B3: UI remains read-only and uses `chunk_id` for every deep link.

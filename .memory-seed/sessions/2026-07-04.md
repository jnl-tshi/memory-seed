### Summary

- Resolved the `related_degree` naming collision flagged during ranking P1a. User's call: rename the Lense field to `connectivity`, and rename the backend field to something descriptive of what it measures. Established (before editing) that the two are genuinely different metrics — Lense's counts combined inbound+outbound `related_entries` edges (a node-sizing connectivity weight), the backend's counts inbound only (the directional importance/attention precursor) — so this is a two-way rename to distinct names, not "keep the newer one."

### Decision

- D: Lense's graph-node `related_degree` → `connectivity` (computation unchanged: `_related_degrees()` → `_connectivity_degrees()`, the `_graph_node` param/dict key, and the `app.js` node-radius consumer). Backend's inbound-only signal `related_degree` → `inbound_relation_count` on `link show` and `memory_get_chunk`. Both renames complete across code, frontend JS, and tests.
- R: The names now describe what each computes. `connectivity` reads as a graph-topology display weight; `inbound_relation_count` reads as "how many entries reference this one," which is exactly `len(node.inbound)`. Renaming was free of external-contract risk: the backend field was unreleased (ranking P1a, unshipped), and the Lense field, though shipped in 2.13.0, is a local read-only UI whose frontend ships in lockstep with the backend — no third-party consumer.
- A: Considered aligning Lense to the inbound-only definition instead (one metric everywhere). Rejected: Lense genuinely wants the combined-degree number for node sizing (a node connected in either direction should draw larger), so collapsing to inbound-only would degrade the visualization to serve a naming convenience. Keeping two well-named metrics is correct.
- F: `memory_seed/cli.py`, `memory_seed/mcp_server.py`, `memory_seed/lense.py`, `memory_seed/lense_static/app.js`, `tests/test_mcp_server.py`, `tests/test_lense.py`, `CHANGELOG.md` (P1a bullet reworded + new Lense-rename bullet), `docs/todo/interaction-frequency-ranking-plan.md` (P1a heading/body + collision note now marked resolved + DoD checkboxes), `docs/todo/NEXT_STEPS.md`, `docs/functionality-audit.md` (§14 refreshed: 5 of 6 logic-capture items now implemented, not 2).

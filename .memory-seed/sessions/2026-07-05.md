## 2026-07-05 11:30 - Scope session decision diagrams: two-class model + sidecar convention

```yaml
entry_id: mse_w5nfex8b4bdmn438
user_initials: JNL
agent_type: claude
agent_name: null
project_path: .
subproject_path: null
related_entries:

  - mse_7grk25jt7wn86a92
```

### Summary

- Turned a discussion about whether session logs should use Mermaid into a scoped plan. The trigger:
  no session entry has ever embedded a diagram (25 files), which is fine for technical readers but
  weak once Memory Explorer becomes a paid non-technical review tool. Wrote
  `docs/todo/session-decision-diagrams-plan.md` and wired it into the Explorer distribution plan and
  roadmap surfaces.

### Decisions

#### D1 - Two-class diagram model; reasoning diagrams are authored sidecars, not inline, not render-derived

- D: Class 1 (structural) — provenance/timeline/graph views the no-LLM Explorer derives
  deterministically from `build_related_entry_graph()`, plus a new exportable static report/handover
  pack (the paid deliverable, which does not exist yet — only the interactive Lense views do). Class 2
  (reasoning) — per-decision flow/sequence diagrams the agent authors at entry time as
  `.memory-seed/sessions/diagrams/<entry_id>.md` sidecars, kept out of the append-only log, validated
  by `check_session_links` (`orphan-diagram`/`diagram-filename-mismatch`/`malformed-diagram`), rendered
  beside their entry.
- R: A no-LLM, deterministic, local-first Explorer cannot turn freeform prose reasoning into a
  flowchart — so a reasoning diagram must be produced by the agent (the LLM) in the same turn it writes
  the entry. Inlining Mermaid in the log pollutes the diffable audit trail (the differentiator) and
  never moved the base rate anyway; render-time derivation is impossible for this class. The sidecar
  resolves both: authored when the LLM is present, stored out of the prose log, and legitimately frozen
  because it documents the decision as made (immutability is a feature here, unlike a roadmap view).
- A: Considered a stored `diagrams:` field on the entry YAML for forward-linking — rejected to keep a
  single authoritative link source (the sidecar's `entry_id` frontmatter) and avoid touching the
  append-only entry. Considered raw `.mmd` with frontmatter — rejected because YAML frontmatter breaks
  Mermaid parsers; used a `.md` wrapper (frontmatter + fenced mermaid) instead, which also lets
  `check_session_links` read it the same way it reads entry YAML. Named the anti-empty-folder trigger
  explicitly (an opportunistic, gated authoring rule in `session_logging.md`/`end_of_turn.md`) since a
  storage spec alone would sit empty like inline Mermaid did; and stated plainly that Class-2 coverage
  is sporadic, so Class-1 derived views + the report pack are the uniform stakeholder backbone.
- F: `docs/todo/session-decision-diagrams-plan.md` (new), `docs/todo/memory-seed-explorer-distribution-plan.md`
  (Phase 1 sidecar-surfacing bullet, Phase 2 report-pack note, provenance link),
  `docs/todo/NEXT_STEPS.md` (specs entry), `docs/functionality-audit.md` (proposed-item entry).
- T: Docs-only (no code yet — the convention/validation/guidance are specified for a later
  implementation increment). `links check` clean (25 files); `doctor` healthy.

### Follow-up

- Phase 1 (core: `sessions/diagrams/` convention, `check_session_links` issue kinds, authoring
  guidance in `session_logging.md`/`end_of_turn.md` + seed twins) is the next implementable increment
  and is Explorer-independent. Phase 2 (Explorer render) rides the distribution plan's Phase-1

- D: A diamond badge in the Trail, or a diagram figure in the reader, opens a modal viewer with

- R: JNL asked for parity with the vanilla JS. The React badge was an inert `<span>` whose tooltip

  unreadable in the React client with no way to enlarge it. Reusing vanilla's class names rather than
  inventing new ones means one stylesheet change tracks both UIs instead of them drifting.
- A: The badge fetches the entry's own chunk rather than requiring selection first, because the point
  of a preview badge is to look WITHOUT navigating away from what you are reading; it reuses the
  loaded chunk when it is already the selected entry. Zoom is extracted to a pure module for the same
  reason as graphLayout/graphEdges: the arithmetic (clamping, fit-scale, zoom-about-cursor) is where
  the bugs are and none of it needs a DOM.

- D: The viewer's bad first render was a MISSING CSS RULE - `.diagram-stage .diagram-svg { width:
  100%; height: auto; }`, which I dropped when porting the vanilla styles. Restored.
- R: Seeing the modal render nodes strung out in a wide unreadable row, I concluded the diagrams
  themselves were at fault: `arc2d` has genuinely zero `subgraph` support (verified), and my sidecars

  is unsupported by the renderer" was a tidy story. JNL said plainly that the inspector renders the
  diagrams correctly - the same diagrams, the same renderer - which falsified it in one line. The
  difference was never the source; it was that the inspector's inline `.diagram-svg` sizes with

- A: I had already rewritten the three diagrams before the correction. They are kept, because the
  subgraph finding IS true and independently worth acting on - `arc2d` cannot draw the construct the
  skill tells authors to use, so a subgraph diagram silently loses its grouping in both UIs. That is a

  fixed in this entry.

  corrected by someone who checked a second surface. The cheap test I skipped was comparing the
  inspector and the viewer on the same diagram before theorising.

### Decisions

#### D1 - Semantic similarity ranks link-audit candidates; it never decides membership

- D: `audit_link_gaps` adds `SEMANTIC_OVERLAP_BOOST * cosine(source, candidate)` to the existing
  lexical score, using model2vec embeddings of each entry's full text. The lexical gate (shared
  files, shared title terms, or shared topics when unlinked) still decides which pairs are candidates
  at all; semantic similarity only reorders what got through.
- R: Measured against the 104 supersedes/evolves edges authors declared in entry YAML, recall@5 goes
  60% -> 77% and recall@10 64% -> 82%, with 85/104 -> 97/104 true targets surfaced at all. That is a
  large gain for a term that costs 0.15s to compute across 544 entries.
  The gate/rank split is not a stylistic choice: cosine is DENSE. Every pair of entries has non-zero
  similarity, so using it to decide candidacy would make every earlier entry a candidate for every
  later one - the all-pairs explosion the live spec feared, arrived at from the opposite direction.
  Ranking has no such problem because the candidate set is already bounded by the lexical gate.
- A: Fails open. `_embed_entries_for_link_audit` returns None on ANY failure - no provider, no cached
  model, an embed exception - and the score degrades to lexical rather than raising. That path is not
  hypothetical: `pip install --no-deps memory-seed` is the documented lightweight install, and it is
  now tested explicitly by blocking the model2vec import and asserting link audit still returns
  results with finite scores.

#### D2 - The weight is a measured peak, and measured through the shipped code

- D: `SEMANTIC_OVERLAP_BOOST = 160.0`.
- R: Swept 0/5/10/20/40/80/160, then 160/320/640/1280 because recall was still climbing at the top of
  the first range. recall@5 peaks at 160 (77%) and falls away to 72% by 320 and 62% by 1280, so 160
  is a real optimum rather than the largest value tried. An earlier standalone estimate of ~120 came
  from a reimplementation of the scorer and is superseded by this end-to-end number.
  The weight is large only because the terms are on different scales: cosine is bounded to [0,1]

- D: An inferred colour is `pastelOf` the blended community colour - same hue, lifted light,

- R: The two are different operations and only one of them is what JNL asked for. Blending toward a

  the theme dependency from the inference entirely.
- A: `pastelOf` moved from `trailModel.ts` to `colour.ts` and is re-exported, so the Trail's D2..DN

- D: The colour is a weighted OKLab mean across every community voting for a node; strength is

- R: JNL's spec, replacing majority-wins: "the more topics of a given type link to an entry the

  because gamma-space averaging of two hues produces the muddy too-dark midpoint.
- A: The 0.8 cap is the honesty rule restated: however many neighbours agree, an inferred node must
  never render identically to an authored one.

- D: Residue travels down chains of topicless entries, decaying by 0.35 per hop, hard-stopped at 3

- R: JNL relaxed the one-hop rule I had pinned with a test ("inference never propagates"). The

  MEMBERSHIP; this is a bounded visual diffusion that never claims membership, and the decay plus
  hop cap guarantees it dies out instead of flooding a component. Classified nodes never receive
  residue (they have their own colour) and never relay it (their colour already speaks one hop into

- R: The seeding constraint is JNL's precise wording - "has to start from a topic to no-topic

- A: The test that pinned the old rule was rewritten, not deleted: the new suite pins decay

- A: Same-hue-different-lightness pairs are less separated than the measured base palette. Accepted

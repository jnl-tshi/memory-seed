### Summary

- Researched the React Trail's hand-drawn line effect and compared procedural spline geometry, SVG turbulence, Rough.js, D3 curves, and variable-width freehand strokes.

### Validation

- Confirmed `TrailWorkspace.tsx` currently draws lane segments as vertical SVG `<line>` elements and applies one shared `feTurbulence` displacement filter; this changes surface pixels but not the ruler-straight, parallel path geometry.

### Follow-up

- Recommended a deterministic, branch-seeded polyline-to-cubic-path generator with low-frequency lateral drift and constrained endpoints, retaining a much weaker SVG displacement only as surface texture.
  - mse_c7t7a609yvr1bzgy
```

### Decisions

#### D1 - Move the hand-drawn character from the filter into the path geometry

- D: New pure module `trailPath.ts` generates deterministic, smoothly curved SVG paths for the Trail
  rail: interior control points every ~65px, lateral drift capped at 1.1px from two seeded sinusoids
  whose amplitudes sum to 1 (so the bound holds analytically, not by sampling), smoothstep-tapered to
  zero within 18px of each endpoint, splined through Catmull-Rom-to-Bezier at tension 0.45. Seeds are
  FNV-1a over `branch + ":" + role`, so geometry is stable across renders and distinct per branch.
  Lane segments, trunk and both connector legs now route through it; the rounded fork/merge elbows
  stay geometrically exact between drifted legs.
- R: The old "Drawn" mode displaced pixels under a shared `feTurbulence`, which leaves the underlying
  lines straight and parallel - neighbouring lanes shifted *together* and read as offset copies of one
  line, which is the noise JNL reported. Drift has to live in the geometry for lanes to change angle
  independently.
- A: Rejected Rough.js / perfect-freehand / D3 - a ~90-line local helper covers this and the rail must
  stay dependency-free. Rejected pinning the spline to interior dot rows: that forces drift to zero
  every 30px and rebuilds the nervous polyline the brief rules out.
- F: `memory-trace/client/src/trailPath.ts` (new), `TrailWorkspace.tsx`.
- T: `memory-trace/client/src/trailPath.test.ts` (new, 15 cases on node:test with native TS stripping -
  no runner dependency added): determinism, per-seed distinctness, exact endpoints, drift bound, lock
  zone, low-frequency correlation, spacing band, short-span degradation, Slick exactness.

#### D2 - Coalesce per-row lane segments into one stroke per branch

- D: A branch's lane is now a single `<path>` from its first to its last row, replacing the per-row-pair
  `<line>` elements.
- R: The old segments were collinear and shared endpoints, so their union was already one vertical line
  - but a 30px segment is shorter than the control-point spacing and would degrade to straight every
  time, so nothing longer than a row could ever curve. Drawing the whole run as one path is what makes
  a long branch read as one confident stroke; the longest measured in the live corpus is 5641px.
- F: `TrailWorkspace.tsx`.
- T: verified in-browser - rail geometry byte-identical across selection, re-selection and resize.

#### D3 - Demote turbulence to sub-pixel grain, tuned against the magnified rail

- D: `baseFrequency` 0.007 -> 0.09, displacement `scale` 7 -> 0.9, Gaussian blur removed.
- R: `feDisplacementMap` offsets by +/- scale/2, so the old scale 7 moved edges up to 3.5px - on a 1.8px
  stroke that is a second, competing wobble, and at 0.007 its wavelength was long enough to shift whole
  lanes coherently. Now the spline owns the character and the filter only roughens the edge within half
  a pixel. An intermediate 0.03/1.8 was tried and rejected on inspection at 3.2x magnification: it
  visibly rippled the stroke, reproducing the jaggedness JNL flagged earlier in the Drawn style.
- F: `TrailWorkspace.tsx`.

#### D4 - Fix pressed-control contrast and native scrollbars in the React workspace

- D: `color-scheme` declared per theme plus warm `scrollbar-color`/`::-webkit-scrollbar` tokens; dark
  `--on-accent` flipped from near-black `#1d1811` to pale `#f2ece0`; `.dock-control button[aria-pressed]`
  given the `--accent-chip-bg` surface it always assumed; `.topic` hover split from `.topic.active`.
- R: The client never declared `color-scheme`, so the UA painted light scrollbars over warm charcoal -
  JNL's "bright white and grey". The pressed dock button set accent *ink* without an accent *surface*,
  so it inherited `--chip`: that is JNL's unreadable black lettering in dark (1.4:1), and the same rule
  was near-invisible in light too (cream on cream, 1.15:1) - a bug the report did not name. Dark
  `--on-accent` was a dark ink for a bright surface, but the pressed surface in dark is dark teal;
  lightening the ink mirrors light mode's medium-surface/light-ink pairing rather than brightening the
  chip into a loud pill. `.topic:hover` used `--on-accent` with no accent surface, invisible in both
  themes.
- F: `memory-trace/client/src/styles.css`.
- T: measured in-browser - pressed dock and all pressed segment controls 1.4:1 -> 5.96:1 (dark) and
  1.15:1 -> 4.68:1 (light); both now pass WCAG AA.

### Verification

- T: client `typecheck` + `build` clean; 15 new client unit tests pass; 150 Memory Trace tests pass;
  592 core tests pass. In-browser on the live corpus (fresh asset hashes confirmed): max lateral drift
  1.08px, all 273 dots sit exactly on a lane path anchor, 0 endpoints off-lane, rail stable across
  re-renders, Slick emits the original exact coordinates and drops the filter.
- Note: `tsconfig.json` now excludes `**/*.test.ts` so the existing typecheck gate ignores node
  builtins; wiring `npm test` into `verify.yml` is left for JNL (control-plane change).

- R: JNL asked for stability between neighbouring selections and directional, accelerating travel
  otherwise. Native `scrollIntoView({behavior:"smooth"})` can target neither a specific band edge nor
  scale its duration, so a small rAF pass is the honest implementation rather than gold-plating.
  Row geometry comes from the model (`index * TRAIL_ROW + TRAIL_ROW/2`), not DOM measurement, since
  the rail already shares those coordinates.
- A: Rejected `block:"center"`: recentring is what JNL called "very annoying" earlier, and it moves

#### D2 - A new corpus opens at the top

- D: Loading Trace or switching worktree leaves the Trail at row 0: the reset is keyed on the `trail`

- R: The root cause was not a missing scroll reset - `loadGraph` auto-selects a node, and the scroll
  effect then chased it, parking a fresh corpus at whatever offset that entry happened to have

- F: `TrailWorkspace.tsx`.
- T: verified `scrollTop == 0` on reload with a selection already present.

#### D3 - Floating find bar over full-corpus matches

- D: A dismissable Ctrl-F-style bar (bottom-right of the Trail) with a match counter and prev/next

  Enter/Shift+Enter cycle with wraparound, Esc dismisses, a new query re-opens it. Matching runs over
  the **whole corpus** (`entryIndex`), and Enter falls through to the server full-text search only
  when the Trail has no local match.
- R: JNL asked for per-candidate investigation like a document Ctrl-F and chose the floating surface.
  Matching only the loaded window would make "next" unable to reach un-paged entries - the one thing
  a find must do; jumping to an off-window match reuses the existing grow-then-scroll two-pass.

- A: Bottom-right, not top-right: the Trail's view bar wraps its legend at narrow widths, and a

#### D4 - Inspector metadata reflows with the pane, not the viewport

- D: Metadata regrouped (identity, provenance, counts, topic chips) into a grid driven by a CSS

- R: JNL's only firm requirement was that it be aware of pane width. The inspector is user-resizable

  measuring the wrong box. Authority and provenance stay on separate lines: BG1 keeps them as

- F: `App.tsx`, `styles.css`.
- T: measured reflow 1/1/2/2/3 columns at pane widths 270/320/420/540/620, and 3 columns at 778px

#### D5 - Pen pressure as a variable-width ribbon, plus live tuning dials

- D: `ribbonEdges`/`ribbonPath`/`pressurePath` build a closed, variable-width outline around the

  mode only. The settings menu regains sliders for **wobble** (turbulence displacement) and

- R: SVG `stroke-width` is constant per path, so real thickness variation needs a filled outline.
  Keeping the centerline untouched means every geometry guarantee from `mse_6bmkhqwaax1wh0z3`
  (exact endpoints, drift bound, determinism, Slick byte-exactness) still holds unmodified, and the
  endpoint taper is what lets a ribbon meet a constant-width fork elbow without a step. Runs too

  Dials are live so JNL can settle values before they are baked, as with the previous tuning pass.
- A: Considered layered strokes (cheaper) - rejected, they cannot vary width *along* a stroke, only

- F: `trailPath.ts`, `TrailWorkspace.tsx`, `styles.css`.

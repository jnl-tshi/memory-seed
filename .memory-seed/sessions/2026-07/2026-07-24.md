### Decisions

#### D1 - Nested fields need nested restoration, and the storage boundary owns it

- D: `_chunk_from_storage` restores `decision_edges` as a tuple of tuples, spelled out separately from the flat `tuple(...)` loop that handles every other collection field. The fix goes at the deserialization boundary rather than at the consumer that crashed.
- R: JSON has no tuple, so every collection field arrives as a list; the flat loop exists precisely to undo that, and `decision_edges` was simply never added to it when the field landed this morning. Fixing it at the crash site (`dict.fromkeys` in `graph`) would have made that one caller defensive while leaving every future consumer holding unhashable lists - the field is a set member and a dict key by nature. Nested is the part that bites: appending the key to the existing list would have restored the outer tuple and left each edge a list, which fails identically.
- A: Considered making the field a frozen dataclass like `ContinuityBlock`, which the same function already reconstructs properly - the right shape long-term, and worth doing if a fifth element is ever added, but a type change is not what a red UI needs.
- F: `memory-trace/memory_trace/service.py`.
- T: `memory-trace/tests/test_trail_decision_edges.py` - reverting the fix reproduces the exact live `unhashable type: 'list'`.

#### D2 - The suite could not see this, so the new test loads from storage rather than rebuilding

- D: The regression test constructs a second `TraceCache` over the same `cache_root` without rebuilding, asserts the edges come back hashable, and renders the Trail through it.
- R: Every existing Trail test calls `cache.rebuild()`, which extracts chunks fresh in-process - where the extractor's tuples are still intact - so none of them ever exercised deserialization. The equivalence suite compares `json.dumps` of each chunk, and a tuple and a list serialize to identical bytes, so it was structurally blind too: the corpus was fine, the projection was fine, and only a *second process* reading it back could fail. Every test passing while the live UI is red is the signal that the tests share a blind spot with each other, not that the bug is exotic.
- A: Considered asserting on `_chunk_from_storage` directly - narrower and faster, but it would not have caught the consumer-side crash, which is what the user actually saw.
- F: `memory-trace/tests/test_trail_decision_edges.py`.

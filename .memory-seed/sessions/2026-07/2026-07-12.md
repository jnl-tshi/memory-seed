## 2026-07-12 15:15 - Trail: commit-accurate merges from Memory-Entry trailers

```yaml

```

### Decision

- D: The Trail's fork/merge connectors now draw from commit ground truth instead of the positional
  heuristic. Backend (`lense.py`): `_first_parent_trailer_commits()` runs one
  `git log --first-parent` pass with a `%(trailers:key=Memory-Entry)` format over the trunk,
  yielding every merge commit that `session merge-branch` stamped with entry trailers, plus each
  merge's fork point (`git merge-base` of its parents); cached as `trailer_merges` meta beside
  `entry_commits`. `graph()` gains `merges` (ordered events intersecting displayed entries) and
  `branches` (per-branch `merge`/`fork`/`estimated`); `chunk()` gains `merged_by`. Frontend
  (`app.js`): a commit is a TIME on the trunk, not an entry row, so anchors interpolate to
  fractional rows between the bracketing entries - back-to-back merges spread by commit time.
  Trunk merge rings render one per event (hover = sha + subject, click selects the merged work);
  connector hovers say "merged by <sha> <subject>" or "merge/fork point estimated"; the reader's
  Commit section splits "Authored in" (diff-derived) from "Merged to main by" (trailer-derived).
- R: Entry heading times can run ahead of the commit clock (today's three merges: entries 13:20/
  13:55/14:45, merge commits 12:51/13:00/13:17), so anchors clamp to stay newer than the newest
  merged entry - a merge drawn below its own content reads upside-down. Branch closure was also
  sharpened during implementation: a branch closes only when its NEWEST displayed entry is merged;
  newer unmerged work leaves it dangling open (accurate) while earlier merges stay visible as trunk
  rings. The entry->commit join lives on the commit side (trailers) by necessity - an entry cannot
  contain the SHA of a commit that hashes over it, which is why write-time stamping into entries

### Summary

- Evaluated the proposed `memory-seed session fuse` workflow before implementation, including Git
  merge primitives, existing session parsing code, and team-use constraints.

### Decision

- D: Treat session fuse as an explicit Memory Seed-aware integration command, not as a Git union
  merge driver or automatic notes mechanism.

- D: Add `session_fuse()` plus `memory-seed session fuse --branch <branch> [--base <ref>] [--apply]`.
- R: Git merge mechanics do not know session-entry immutability, `branch:` provenance, grouped target
  paths, sidecar parent requirements, or chronological session ordering.

#### D2 - Require sidecar parent acceptance

- D: Import diagram sidecars only when the parent entry is already present on the base/main tree or
  the parent branch entry is accepted for promotion in the same fuse.
- R: A sidecar without a promoted parent creates an orphan reasoning artifact and breaks the
  entry-linked diagram contract.
- A: Malformed sidecar blocks without `entry_id`, orphan sidecars, modified existing sidecars, and
  date mismatches now block fuse planning.
- F: `memory_seed/core.py`, `tests/test_memory_seed.py`.

#### D3 - Add MCP and skill surfaces

- D: Expose `memory_branch_status` and `memory_session_fuse_preview` as read-only MCP tools, and route
  their usage through `agent_collaboration.md`.
- R: The intended operator is often an LLM; it needs structured branch/fuse state and skill guidance,
  while file-changing apply remains an explicit CLI action.
- A: `history_retrieval.md` now describes retrieval MCP tools only; `agent_collaboration.md` owns the
  branch/fuse MCP control surface.

### Decision

- D: Made the Graph Workspace's "Edges" filter row (Related/Replaces/Evolves/Topic) a pure
  client-side visibility toggle, at JNL's direct request to take inspiration from Obsidian's graph
  view: lines appear and disappear, but the node set and layout never move unless a genuine
  node-specific filter (scope, topic, date range) is applied.
- R: Explained to JNL first why toggling a chip previously made the whole map jump — each click added
  or removed that edge type from the `edge_types` query param and re-fetched
  `/api/v1/graph/projection`, which could change the returned node set (a node connected only via the
  toggled-off type simply vanished) and always caused `GraphWorkspace`'s mount effect to re-run:
  a fresh Cytoscape instance, new deterministic starting positions for whatever node set came back,
  and a from-scratch 900-iteration `cose` layout. Selection was already handled in place with no
  remount (`GraphWorkspace.tsx`'s "mount ONCE per graph payload" comment); the fix was to give the
  edge-type filter the same treatment instead of coupling it to the fetch. One wrinkle found while
  implementing: the filter row is hidden in Evolution/File scope, but `edgeTypes` state persists
  across scope changes — without a guard, turning "Evolves" off in Overview and then switching to
  Evolution scope would silently hide every edge in a scope whose entire purpose is showing the
  evolves/supersedes chain, with no visible chip to explain why. Fixed by having Evolution/File scope
  always pass their own fixed edge-type set as `visibleEdgeTypes`, ignoring whatever the (hidden) row
  is currently set to.
- A: Added `edgeTypesForScope(targetScope)` (`App.tsx`) so every fetch site (`load`, `requestGraph`'s
  default, `changeScope`/`openFileMode`/`chooseTopic`/`changeRange`) derives its edge-type query param
  purely from scope, never from the filter-row state; `toggleEdge` is now synchronous and only calls
  `setEdgeTypes`, no `requestGraph`. `GraphWorkspace` takes a new `visibleEdgeTypes` prop, applied in
  the existing `applyPresentation` in-place pass alongside selection/labels: an `edge-filtered` class
  (mirroring the existing `hidden-until-selected` mechanism for evolves) toggles `display: none` per
  edge type without touching nodes, positions, or the layout. The presentation effect's dependency
  array picked up `visibleEdgeTypes`; the mount effect's deps (`[graph, renderedNodes, theme]`) are
  unchanged, so it genuinely never re-runs on a chip click. `App.tsx` passes
  `scope === "evolution" || scope === "file" ? edgeTypesForScope(scope) : edgeTypes` as
  `visibleEdgeTypes` for the scope guard above.
- F: `memory-trace/client/src/App.tsx`, `memory-trace/client/src/GraphWorkspace.tsx`, rebuilt
  `memory-trace/memory_trace/static/react/`.
- T: `npm run typecheck` - clean. `npm test` - 69/69 unchanged. `npm run build` - succeeds.
  Live-verified against the real packaged app's `/next` React shell via the Cytoscape debug harness

  `openContextEntry`.
- A: `openContextEntry` in `App.tsx` now does `graph?.nodes.find((item) => item.source.entry_id ===
  entryId)` first; a hit calls `select(node)` and returns, a miss falls through to the existing

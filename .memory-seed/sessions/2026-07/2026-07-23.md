## 2026-07-23 08:25 - Diagram viewer: click to zoom, scroll to move, fit stays centred


### Summary

- Two fixes to the multi-diagram modal viewer, both from the wheel doing one job too many. Scrolling
  a multi-diagram view now moves between diagrams; a diagram zooms only after you click it; and the
  fit stays centred across resizes. All from JNL's report.

### Decisions

#### D1 - The wheel scrolls until a diagram is clicked, then zooms

- D: A viewport claims the wheel only once ACTIVE (a click). Inactive, the wheel scrolls the modal
  body between diagrams. An accent ring plus a "Click to zoom" hint signal the state; Escape releases
  an active diagram before it closes the modal.
- R: Every viewport ate the wheel and zoomed, so you could not scroll from one diagram to the next -
  you were stuck zooming whichever one the pointer was over. Gating on an explicit active state gives
  the wheel back to the list until you ask for zoom.
- A: One active diagram at a time, held in the parent, so activating one releases the rest. A press
  activates and arms a drag in the same gesture; panning waits until active.

#### D2 - A native, non-passive wheel listener, because React's onWheel cannot preventDefault

- D: The wheel is attached with `addEventListener("wheel", ..., { passive: false })` on the viewport
  ref, not through React's `onWheel`.
- R: React registers its delegated root wheel listener as PASSIVE, so `event.preventDefault()` inside
  `onWheel` is silently ignored. Measured live: an active diagram zoomed but `defaultPrevented` stayed
  false - it would have zoomed AND let the body scroll underneath at the same time. The original code
  carried the same latent no-op; gating the zoom is what made it bite. A native non-passive listener
  is the only place preventDefault actually holds.
- A: The listener reads `active` through a ref, so it attaches once and never has to re-bind when the
  active diagram changes.

#### D3 - Fit stays centred through a resize, then yields to the reader

- D: A ResizeObserver re-fits the diagram while the view is still "just fitted", and stops the moment
  the reader zooms or pans (an `adjusted` ref). "Fit" re-centres and clears the flag.
- R: The fit maths already centred on both axes; what was missing is that a viewport resize (window,
  the inspector dock moving) never re-ran fit, so the diagram could sit off to one side. Re-fitting
  only while untouched keeps it centred without ever yanking a view the reader has framed themselves.

### Validation

- Live, on the 2-diagram 23:04 entry: inactive wheel does not zoom and does not preventDefault (body
  free to scroll); a click adds `.active` and hides the hint; active wheel zooms (scale 0.589 ->
  0.660) and preventDefault holds (body locked).
- Centred on fresh open (both diagrams, x and y), after "Fit", and across a 1149 -> 675 -> 1149
  viewport resize.
- 134 client unit tests pass, including new assertions that `fitTransform` centres on the downscale
  path; typecheck and vite build clean; no console errors.

### Follow-up

- The wheel gating, activation and resize-refit are DOM behaviours verified live; there is no React
  component-test harness here (only `diagramZoom` pure-maths tests), so those three are not unit
  covered. A harness would be its own piece of work, not a rider on this fix.

### Summary

- A pre-commit hook now parses every staged diagram sidecar through real Mermaid and blocks a commit
  that would ship an unparseable diagram - the backstop `links check` cannot be, because it validates
  a sidecar's structure and never parses the Mermaid. JNL asked for it after the two published breaks.

### Decisions

#### D1 - The check needs a DOM, because parse runs HTML labels through DOMPurify

- D: The checker sets up jsdom (a dev-only dependency) before importing mermaid, then calls
  `mermaid.parse` on each ```mermaid block.
- R: A first spike parsed a trivial `flowchart TD; A-->B` in bare node and I nearly concluded parse
  needs no DOM. The full corpus then failed 54/56 with "DOMPurify.addHook is not a function": every
  real sidecar uses `<br/>`, and mermaid runs HTML labels through DOMPurify, which needs a `window` to
  initialise. jsdom set before the import fixes it; the broken cases still fail correctly.
- A: `render()` is never called, only `parse()`, so this stays a lint, not a headless renderer -
  jsdom is enough, no puppeteer/Chromium. Measured: 56/56 parse, and a `;`-in-message block still
  fails on line 2.
- A: The trivial spike was a false green - it did not exercise the `<br/>` path the whole corpus uses.
  The lesson is the same one as the flowchart.htmlLabels false-positive earlier: a probe that does not
  exercise the real input can pass while the thing it claims to prove is false.

#### D2 - Best-effort hook, reusable checker, not a seeded requirement

- D: The hook skips cleanly when node or the client's mermaid/jsdom are absent (exit 0, never blocks);
  `check-sidecar-mermaid.mjs` exports `checkSidecars()` and also runs as a CLI; `install-git-hooks.mjs`
  writes the pre-commit shim idempotently and refuses to clobber a foreign hook.
- R: Mermaid is a Trace dependency, not a seed one, so a mermaid-parsing hook cannot be a mandatory
  part of `memory-seed hooks install` (which owns only prepare-commit-msg) without breaking the seed's
  lightweight, dependency-free posture. A repo-local, self-skipping hook is the right shape.
- A: The exported `checkSidecars` is deliberately the seam the planned CI parse-test reuses, so the
  parser is wired once, not twice. `npm run lint:sidecars` is the manual/CI entry.

### Validation

- End to end: a staged broken sidecar aborted the commit with file + block + parse error, HEAD
  unchanged; a commit with no staged sidecars passed (this feature's own commit).
- `npm run lint:sidecars` -> 56 blocks parse across 8 files; typecheck clean; 134 client unit tests
  pass; `npm run hooks:install` wrote `.git/hooks/pre-commit`.

### Follow-up

- The CI parse-test task can now shrink to "call `checkSidecars` / `npm run lint:sidecars` in CI" -
  parse-headless-with-jsdom is verified and the checker exists. The chip is updated.
- jsdom is a new client devDependency; it ships in no runtime bundle (`render()` is never invoked).

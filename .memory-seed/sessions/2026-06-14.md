### Summary

Did the demo fix now and built the generalized CLI feature ("do both").

### Decision

- D: Added a 4-way ownership branch (shared `_maybe_merge_foreign_routing` used by both `init_project` and `update_project`) over the four `ROUTING_DESTINATIONS` (`AGENTS.md`/`CLAUDE.md`/`GEMINI.md`/`.github/copilot-instructions.md`): (1) absent → write full seed file; (2) ours (`memory-system-version` frontmatter) → version-gated archive+replace (unchanged); (3) foreign + our `<!-- BEGIN memory-seed -->` markers → re-sync block in place; (4) foreign, no markers → inject block, append at end. A foreign file is **never** overwritten, even under `init --force`. `_merge_routing_stanza` mirrors `_merge_grouped_hook`. Added a `doctor` route-presence warning (runtime exists but a present entry-point file is foreign-without-block → orphaned runtime) and made `doctor` skip foreign routing files from version-mismatch. Injected the block into `demo/AGENTS.md` + `demo/CLAUDE.md` (untracked). Bumped control-plane 2.7→2.8 / package 2.8.0.
- R: Ownership is frontmatter-only and fail-safe to merge (advisor-confirmed): when we cannot prove a file is ours, appending a block is non-destructive; overwriting is the bug being removed. This retires the legacy "versionless → archive `unknown-*` + clobber" path. The user explicitly asked the CLI to check ownership *before* archiving-or-merging, and to think through the "second merge" on a later version bump — handled by in-place block replacement gated on **body** content-equality.
- A: Signature/content-guessing for ownership (rejected — reintroduces overwrite-on-guess); version-stamping the block (rejected mid-build — the advisor caught that `v=2.8` inside the compared block would force a rewrite on every bump, contradicting the no-churn claim; removed the stamp so a foreign, non-version-tracked block re-syncs only on body change).
